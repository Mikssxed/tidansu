Known-open gaps — check status each review rather than re-discovering:

- **Auth rate limit tightening — PARTLY LANDED (as of B-7, 2026-07-13).** A dedicated `magic-link` policy (3/min per IP, `WebApplicationBuilderExtensions.cs`) now sits alongside the shared `auth` policy (10/min), and a per-recipient `MagicLinkThrottle` (in-memory, `ServiceCollectionExtensions.cs`) exists to cap sends per address. The residual is deploy-time: the per-IP partition is only meaningful once the trusted proxy is configured (see next bullet). **How to apply:** the Phase-2 per-recipient throttle has landed — verify it's actually applied on the magic-link handler before re-flagging the mailbomb vector as open.

- **`UseForwardedHeaders` — config-driven (`Program.cs:41-88`).** Binds `KnownProxies`/`KnownNetworks` from config; only explicit entries added, blank→loopback-only default. IPv6 wildcard gap from B-7 is **FIXED as of B-8 (2026-07-14)**: guard now rejects `"*"`, `0.0.0.0/0`, `::/0`, `::`. **Two residual gaps remain:** (1) still misses split full-coverage pairs / `PrefixLength==0` spelled differently (best fix: reject parsed `PrefixLength==0` instead of string-matching); (2) the real trusted proxy address is an **open deploy step** — until set, all users share one rate-limit bucket. **How to apply:** on any rate-limit/proxy change, confirm the real proxy address is configured; the IPv6 string-wildcard case is already handled.

- **Fail-loud guards scoped to `IsProduction()` only (RECURRING — re-confirmed B-7).** JWT-secret, and now (B-7) `AppSettings:FrontendUrl` and `ConnectionStrings:TidansuDb` guards fire only when env == exactly `"Production"`. The SMTP guard fires for all non-Development envs — inconsistent. A mis-named internet-facing env (`Staging`, typo) boots half-broken past these guards. Mitigating: `ASPNETCORE_ENVIRONMENT` unset defaults to Production (bare deploy is safe), and dev shortcuts stay `IsDevelopment()`-gated so no auth bypass opens. Recommend `!IsDevelopment()` scoping. **How to apply:** on any new fail-loud guard, check it uses `!IsDevelopment()` (or an env allowlist), not `IsProduction()`, so Staging isn't a hole.

- **Fail-loud config guards miss the security-critical key.** B-6's Stripe `IsConfigured`/startup guard checked `SecretKey`+`ProPriceId` but omitted `WebhookSecret` — the one bearer credential gating the anonymous webhook (empty secret = forgeable = free Pro). Pattern: when reviewing any `IsConfigured`/fail-loud guard, enumerate *every* secret the feature depends on and confirm each is in the guard, not just the "obvious" ones.
  **How to apply:** on any new config guard or billing/auth secret, check the signing/verification secret is required, not just the API key.

- **Counted plan-caps are checked non-atomically (read-then-insert TOCTOU).** Found B-8 (2026-07-14, S-1): `CreateSpaceCommandHandler` reads `CountByUserAsync` then `AddAsync` in separate DB round-trips with no locking/constraint, so concurrent creates all pass the 2-space cap and over-provision. Zone/item caps are safe (gated on the whole submitted graph in one request); only *cross-request counted* caps race. **CLOSED for spaces by B-12 (2026-07-14):** `ISpacesRepository.AddWithinSpaceCapAsync` now wraps a per-user transaction-scoped exclusive `sp_getapplock` + authoritative in-lock re-count + insert; Pro (null cap) skips the lock. Trust boundaries are clean (resource keyed on server-set `UserId`, `ExecuteSqlInterpolatedAsync` parameterized). **How to apply:** on any new counted-cap mutate path, check count+insert share one serializable tx / app-lock / DB constraint. If a *zone* or *item* cap ever moves to a per-child endpoint (created one at a time), the same TOCTOU re-opens for it — re-flag then.

- **App-lock (`sp_getapplock`) acquisition status must be checked or it fails OPEN.** Found B-12 (2026-07-14, S-J1): `sp_getapplock` signals granted (`0`/`1`) vs failed (`-1` timeout / `-3` deadlock / `-999`) via its **stored-proc return value**, NOT by raising an error. `ExecuteSqlInterpolatedAsync`/`ExecuteSqlRaw` return rows-affected and **ignore the proc return code**, so if the return value isn't captured, a *non-granted* lock (e.g. `@LockTimeout` expiry under contention) falls through to the guarded insert **without the lock held** → the serialization the lock was meant to provide silently disappears (fail-open). **How to apply:** any `sp_getapplock` call must bind an output/return `SqlParameter` and reject `< 0` (rollback + throw) before entering the critical section. A bare `EXEC sp_getapplock ...` with no captured result is a fail-open bug even though it looks correct.

- **Rate limiting is opt-in per policy — non-auth mutate endpoints have none.** Found B-13 (2026-07-15, S-M3): `app.UseRateLimiter()` is wired but only `AuthController`/`BillingController` carry `[EnableRateLimiting]`. `POST`/`PUT /api/spaces` has no policy, and its `[RequestSizeLimit(24 MB)]` body is fully read + JSON-deserialized (multi-MB strings → LOH) **before** the handler's plan gate runs — so even a Free user who will be 403'd costs the server 24 MB per request. **How to apply:** `[RequestSizeLimit]` is a per-request ceiling, never a substitute for a per-user rate limit. On any new mutate endpoint accepting a large body, check for an `[EnableRateLimiting]` policy partitioned on the authenticated user id.

- **Plan-cap "downgrade rule" is count-based, so Pro capabilities survive downgrade.** Found B-13 (2026-07-15, S-M4): `PlanPolicy.CheckSpaceMutation` blocks photos only when `after.Photos > before.Photos`. One month of Pro → N photos → downgrade keeps N photo *slots* permanently, each refillable with brand-new arbitrary images forever (count never rises). "Over-cap content stays editable" reads as read-only in the spec but is implemented as unlimited-replacement. Not a regression; no *unchecked* photo persists (the guard still validates content). **How to apply:** when reviewing any capped *capability* (not just a counted resource), check whether the downgrade rule gates content identity or merely the count.

- **Webhook/event handlers claim idempotency and mutate in separate `SaveChanges`.** Atomic dedupe insert is correct, but a mutation failure after the claim commits strands the operation (Stripe won't retry). Check claim+mutate share one unit of work whenever reviewing at-least-once event handlers.
