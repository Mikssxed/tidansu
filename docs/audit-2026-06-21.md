# Tidansu — Code Audit Report

**Date:** 2026-06-21
**Scope:** Full backend (.NET) + frontend (Vue) sweep, security-prioritized. Branch `main`, clean tree.
**Type:** Audit / findings report only — no code changes made.

**Overall:** This is a high-quality codebase. Clean Architecture boundaries are
respected, handlers are thin, naming is consistent, and the security
fundamentals are genuinely well done. Findings below are mostly hardening and
one real misconfiguration — there are **no critical/exploitable vulnerabilities**
in the data path.

## What's already done right

- **No IDOR.** Every space access is scoped by `userId` at the query level
  (`SpacesRepository.GetByIdAsync(id, userId)`, `GetAllByUserAsync`,
  `CountByUserAsync`). A user cannot read or mutate another user's space even
  with a guessed id.
- **Server-side limit enforcement**, not just client — plan caps are re-checked
  in `CreateSpace`/`UpdateSpace` handlers, with a correct downgrade rule
  (over-cap data stays editable, can't push higher).
- **Tokens hashed at rest** (SHA-256) — magic-link and refresh tokens are never
  stored in plaintext.
- **Single-use magic links**, burned before issuing JWTs, superseded on
  re-request; 15-min lifetime.
- **Stripe webhook signature verified** via `EventUtility.ConstructEvent`
  (forged payload → 400).
- **EF Core throughout** — parameterized, no raw SQL, no injection surface.
- **Good defaults:** security headers (nosniff / X-Frame-Options DENY /
  Referrer-Policy), HSTS + HTTPS redirect in prod, CORS restricted to a
  configured origin (not `*`), `ClockSkew = 0`, full token validation, account
  lockout, prod JWT-secret length guard, and error middleware that doesn't leak
  exception details to clients.

## Security findings

### Medium

**S1 — `VITE_DISABLE_AUTH=true` is committed and bakes into the production bundle**
`src/Tidansu.App/.env:2` → consumed at `src/Tidansu.App/src/router/index.ts:54`.
Because it's `VITE_`-prefixed, `npm run build` (the documented prod build to
`wwwroot`) embeds it, **disabling all client-side route guards in production**.
Impact is contained — backend endpoints are `[Authorize]`, so data stays
protected and API calls 401 — but unauthenticated users can load app shells they
shouldn't. *Fix direction:* move this flag to `.env.local` (gitignored) or
`.env.development`, and assert it's absent in prod builds.

**S2 — JWT access + refresh tokens in `localStorage`**
`useAuthStore.ts:11–42`. Any XSS would exfiltrate both the 60-min access token
and the **7-day refresh token**. Risk is currently low because the frontend is
clean (no `v-html`/`innerHTML`/`eval` anywhere), but it's the highest-leverage
hardening target. *Fix direction:* httpOnly+SameSite refresh cookie is the
hardened pattern; at minimum, document the tradeoff.

### Low / defense-in-depth

**S3 — `/api/auth/refresh` has no rate limit.** `AuthController.cs:43` —
`magic-link` and `consume` carry the `auth` policy; `refresh` doesn't.
Brute-force is infeasible (256-bit random tokens), but add the policy for
consistency.

**S4 — Refresh rotation has no reuse/theft detection.**
`RefreshTokenCommandHandler.cs` rotates correctly but doesn't revoke the token
family when an already-revoked token is replayed. Acceptable;
family-revocation-on-reuse is the hardened upgrade.

**S5 — `returnUrl` isn't validated as a relative path.** Flows from query →
emailed link (`MagicLinkEmailSender.cs:17`) → `router.push(target)`
(`LoginView.vue:187,213`). Practical impact is low (Vue Router won't externally
navigate from a string push, and the victim controls their own `returnUrl`), but
validate it starts with a single `/` as defense-in-depth.

**S6 — Committed dev JWT secret** (`appsettings.Development.json:15`). Clearly
dev-only and the prod path requires `JwtSettings__Secret` via env (guarded at
`WebApplicationBuilderExtensions.cs:26`). Informational — just don't let this
pattern leak to a prod config.

## Clean-code findings

The code is consistent and idiomatic; the store layer (`useSpacesStore.ts`) is
especially clean. Few issues:

**C1 — Missing tests (biggest quality gap).** Zero test projects in the
solution; no frontend unit tests. Security-sensitive logic (plan-limit
enforcement, the downgrade rule, magic-link consumption) is verified only by
manual browser/Node scripts. Highest-value investment for a finished app.

**C2 — Template-purity violation.** `MapZone.vue:18` has an inline ternary in
`:class` (`band.labeled ? '…' : ''`) — a HARD RULE in `CLAUDE.md`. Should be a
`computed`/pre-mapped array. One found; a focused sweep would confirm whether
others slipped in.

**C3 — Known functional gaps** (already noted in the implementation plan, listed
for completeness):
- No automatic 401→refresh interceptor in the Kiota client
  (`useApiClient.ts`) — an expired 60-min access token surfaces as sync errors
  until reload.
- Stripe `StripeBillingService.cs:36` leaves subscription-cancellation-on-downgrade
  and `subscription.deleted` customer↔user mapping as TODOs; the Stripe path is
  built but never key-tested.

## Suggested priority order

1. **S1** (real misconfiguration, easy fix)
2. **C1** (tests — protects everything else)
3. **S2** (token storage hardening)
4. **S3 / S5 / C2** (quick, low-effort hardening + the one style violation)
5. **S4 / C3** (deeper, lower-urgency)
