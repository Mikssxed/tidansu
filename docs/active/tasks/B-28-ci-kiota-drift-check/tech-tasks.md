# B-28 · Tech tasks — CI: fail a PR when the committed Kiota client drifts from a fresh regen

## Structural decisions (delegated to tech-lead — LOCKED here)

- **One workflow file, two parallel jobs** — `.github/workflows/ci.yml` with a
  `build` job and a `kiota-drift` job that run **concurrently** (no `needs:` between
  them).
  - *Why two jobs, not one:* the two concerns fail for different reasons and want
    different diagnosis. A red `build` says "your code doesn't compile"; a red
    `kiota-drift` says "you forgot `npm run build:api`". Separate jobs give the PR
    author two independent, self-describing red/green signals instead of one job
    whose failure could mean either. Parallel wall-clock is also faster than a
    single sequential job.
  - *Why NOT `needs:` (drift depends on build):* GitHub jobs run on separate
    runners with no shared filesystem — a dependency would only serialize them, and
    sharing the compiled API/`node_modules` across jobs would require
    artifact upload/download, which costs more than the duplicate setup it saves.
    Keep them independent.
  - *Accepted cost:* both jobs provision the .NET 10 SDK + Node and both compile the
    API (the `build` job via `dotnet build`, the `kiota-drift` job because
    `npm run build:api` builds `Tidansu.API` to emit the OpenAPI doc). This
    duplication is the price of the clearer signal + parallelism. If CI minutes ever
    bite, collapsing to a single sequential job is the documented fallback — noted,
    not an open question (structure was delegated).

- **Trigger: `pull_request` targeting `main` only** (LOCKED by the human). No
  `push` trigger — direct-push protection is a separate branch-protection admin
  action, out of scope.

- **Drift scope = the committed regen artifacts:**
  `src/Tidansu.App/src/api/apiClient/` **and** `src/Tidansu.App/src/api/api.json`.
  Both are regenerated by `npm run build:api` and both are committed. Diffing
  `api.json` too catches contract/doc drift even in the rare case the emitted client
  files happen to be identical, and gives a more precise "what moved" signal.
  `.kiota.log` is `*.log`-gitignored, so it never dirties the diff.

- **Drift mechanism = `git add -A` + `git diff --cached --exit-code`** (NOT bare
  `git diff --exit-code`). Plain `git diff` does not report **new untracked** files,
  and FR-2 requires failing on added/removed/changed. Staging with `git add -A`
  first makes additions, deletions, and modifications all visible to
  `git diff --cached`. (`.kiota.log` is gitignored so `-A` won't stage it.)

- **Actionable failure (FR-3) via a `::error::` workflow annotation** naming
  `npm run build:api` as the fix and pointing at `CLAUDE.md`, printed alongside a
  `--stat` (not a raw full diff dump).

## Appendix A — `.github/workflows/ci.yml` (starting content)

```yaml
name: CI

on:
  pull_request:
    branches: [main]

permissions:
  contents: read

concurrency:
  group: ci-${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  build:
    name: build (dotnet + spa)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          global-json-file: global.json
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: npm
          cache-dependency-path: src/Tidansu.App/package-lock.json
      - name: dotnet build (solution)
        run: dotnet build Tidansu.sln -c Release
      - name: npm ci
        working-directory: src/Tidansu.App
        run: npm ci
      - name: npm run build (vue-tsc + vite)
        working-directory: src/Tidansu.App
        run: npm run build

  kiota-drift:
    name: kiota drift check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          global-json-file: global.json
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: npm
          cache-dependency-path: src/Tidansu.App/package-lock.json
      - name: dotnet tool restore (pinned kiota)
        run: dotnet tool restore
      - name: npm ci
        working-directory: src/Tidansu.App
        run: npm ci
      - name: regenerate Kiota client (B-21 one-command flow)
        working-directory: src/Tidansu.App
        run: npm run build:api
      - name: fail on client drift
        run: |
          PATHS="src/Tidansu.App/src/api/apiClient src/Tidansu.App/src/api/api.json"
          git add -A -- $PATHS
          if ! git diff --cached --quiet -- $PATHS; then
            echo "::error::The committed Kiota API client is out of date. A backend contract change landed without regenerating the client. Fix: run 'npm run build:api' from src/Tidansu.App and commit the result (see the build:api note in CLAUDE.md)."
            echo "----- drift (git diff --cached --stat) -----"
            git diff --cached --stat -- $PATHS
            exit 1
          fi
          echo "Kiota client is up to date."
```

- Note on `global-json-file`: `actions/setup-dotnet@v4` reads `global.json`
  (`10.0.0`, `rollForward: latestMajor`) and installs a matching .NET 10 SDK even if
  the runner image doesn't ship it yet — see the runner risk in §3.

  - [x] Confirm the trigger is `pull_request`, never `pull_request_target`.

## 3. 📈 Scalability / Correctness Considerations

- **Determinism / no false red (the trust risk).** A flaky drift check trains
  contributors to ignore red. Mitigated by `dotnet tool restore` pinning
  `kiota 1.32.5` (same version as local) and B-21's proven byte-identical idempotence.
  - [x] The local PASS-case verification (two-run empty diff) must hold before merge.
- **New/removed files not caught by bare `git diff`.** `git diff --exit-code` misses
  untracked additions; FR-2 requires failing on added files.
  - [x] Gate uses `git add -A` + `git diff --cached --quiet` so
    additions/deletions/modifications are all detected.
- **`.kiota.log` / non-deterministic artifacts dirtying the diff.** `*.log` is
  gitignored so `git add -A` won't stage `.kiota.log`; the diff is scoped to the two
  committed paths only, not the whole tree.
  - [x] Confirm the scoped `PATHS` list, and that no other regen output lands outside
    those two paths (verify with the local PASS case producing an empty staged diff).
- **CI-minute cost of duplicate API compile across two jobs.** Accepted for signal
  clarity; `cancel-in-progress` concurrency trims superseded runs.
  - [x] Confirm the `concurrency` block is present.

## 5. ❓ Open Questions

1. **Node version pin.** The workflow pins Node `22` (LTS; satisfies Vite 7's
   `>=20.19 || >=22.12`; consistent with `@types/node ^24`). There is no `.nvmrc` or
   `engines` field, so local dev Node is unpinned. Should we add a `.nvmrc` (Node 22)
   so CI and local stay aligned and the workflow can read `node-version-file` instead
   of a literal? Small, durable, but a separate decision — defaulting to a literal
   `'22'` for now.
2. **`.NET 10 SDK on GitHub-hosted runners` (risk, not a blocker).** .NET 10 went GA
   Nov 2025; by mid-2026 `ubuntu-latest` likely ships it, but if not,
   `setup-dotnet@v4` + `global-json-file` downloads a matching 10.0.x SDK — the
   documented, reliable path. Flagged so a first-run "SDK not found" is understood,
   not panicked over. No action unless setup-dotnet itself fails to locate a 10.0.x
   feed.
3. **Required status check on `main` (branch protection).** Out of scope for this
   task (repo-admin action, per the human's LOCKED note). Until configured, the check
   *runs* but does not *block* merge. Confirm the owner will enable it as a required
   check afterward — otherwise FR's "fails a PR" is advisory only.
4. **Scope the drift trigger with `paths:`?** Currently every PR to `main` runs both
   jobs. A backend/frontend `paths:` filter would save minutes on docs-only PRs but
   risks missing an edge case (e.g. a `.config/dotnet-tools.json` bump). Left
   unfiltered for correctness on the first CI; revisit if minutes matter.
