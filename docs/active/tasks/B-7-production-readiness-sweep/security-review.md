# Tidansu — Security Review
**Date:** 2026-07-13
**Scope:** B-7 production-readiness sweep. Four-file diff: `Program.cs` (BA-3 forwarded-headers), `WebApplicationBuilderExtensions.cs` (BA-1 FrontendUrl guard), `ServiceCollectionExtensions.cs` (BA-2 connection-string guard), `appsettings.json` (BA-4 EF Core Serilog level). Plus the adjacent dev-shortcut gates the sweep claims to contain (`EmailService.cs`, `MagicLinkEmailSender.cs`, `router/index.ts`) and the built `wwwroot` artifact.
**Type:** Findings report only — no code changes made.

**Overall:** The diff is sound and does what it set out to do. Every misconfiguration path I traced fails **closed** (refuse to start, or refuse the request), never open to a permissive/dev-equivalent behaviour. No secret is echoed in any new fail-loud message or log line, and the built SPA bundle in `wwwroot` is provably free of the auth-bypass and devtools code. The forwarded-header trust change — the load-bearing one for the rate limiter — correctly *adds* only explicitly-configured proxies, never clears-then-trusts-all, and keeps the loopback-only default when blank. Two residual items are worth the developer's attention: the wildcard rejection is **not airtight for IPv6** (`::/0` slips through), and the fail-loud guards are scoped to `IsProduction()` only, so a mis-named internet-facing environment (e.g. `Staging`) skips them. Neither is a live exploit today given the single-Production launch topology; both are hardening the sweep's own stated goals call for. Nothing in this diff opens another user's data or a plan bypass — deep authz/IDOR stays correctly deferred to B-8.

## What's already done right
- **BA-1 / BA-2 fail loud and name the KEY only.** `AppSettings:FrontendUrl` (`WebApplicationBuilderExtensions.cs:72-76`) and `ConnectionStrings:TidansuDb` (`ServiceCollectionExtensions.cs:27-31`) both throw before `AddCors`/`AddDbContext` in Production with a message that names the env var and echoes **no value**. Mirrors the existing JWT/SMTP/Stripe guards exactly.
- **BA-3 forwarded-header trust is fail-safe.** `Program.cs:44-67` only ever `.Add(...)`s explicitly-configured proxies/networks — there is **no `.Clear()`-then-trust-all path**. Blank config → framework default (loopback only: `::1` + `127.0.0.1/8`), so an arbitrary internet client cannot spoof `X-Forwarded-For` to dodge the limiter. Literal `"*"` (proxies) and `"*"`/`"0.0.0.0/0"` (networks) are rejected at startup with a clear message.
- **Rate limiter partitions on the corrected client IP.** `app.UseForwardedHeaders(...)` runs first in the pipeline (`Program.cs:69`), before `app.UseRateLimiter()` (`:101`). The `auth` (10/min) and `magic-link` (3/min) policies partition on `httpContext.Connection.RemoteIpAddress` (`WebApplicationBuilderExtensions.cs:120,128`), which the forwarded-headers middleware has already rewritten to the real client IP when the proxy is trusted. Ordering is correct.
- **BA-4 hides nothing security-relevant.** Dropping `Microsoft.EntityFrameworkCore` to `Warning` in the base config (`appsettings.json:31`) suppresses only per-query `CommandExecuted` Information chatter; errors/warnings still log. Crucially, `EnableSensitiveDataLogging()` is **dev-gated** (`ServiceCollectionExtensions.cs:36-39`), so production never logged SQL parameter values even at Information — no parameter-value logging remained to worry about, and none is newly hidden.
- **Dev shortcuts confirmed contained — from the artifact, not just source.** (1) Auth bypass: `import.meta.env.DEV && VITE_DISABLE_AUTH === 'true'` (`router/index.ts:55`), and the flag lives **only** in `.env.development` (the build-loaded `.env` contains just `BASE_URL=/`). Grep of the emitted `wwwroot/assets/*.js` for `VITE_DISABLE_AUTH` / `__VUE_DEVTOOLS` / `vue-devtools` is **clean**. (2) Dev email-to-file is `environment.IsDevelopment()`-gated (`EmailService.cs:24`); the prod branch attempts a real send and throws `EmailDeliveryException` on failure — no silent 200. (3) `devLink` is returned only in Development (`MagicLinkEmailSender.cs:24`), `null` otherwise. All three are gated on `IsDevelopment()`, which is off under `ASPNETCORE_ENVIRONMENT=Production`.
- **Email failure path leaks nothing.** On send failure `EmailService.cs:49,58` logs recipient + exception message / provider error only — never the body, the magic link/token, or SMTP creds — then throws `EmailDeliveryException` (sanitized by `ErrorHandlingMiddleware`).
- **No committed secrets.** Base `appsettings.json` carries only blank placeholders; the only JWT secret in source is the clearly-labelled dev value in `appsettings.Development.json` (loaded only when env == Development). `git grep` for `sk_live|sk_test|whsec_|PRIVATE KEY` is clean.

## Security findings

### Critical
_None._

### High
_None._

### Medium

**S-M1 — Fail-loud guards scoped to `IsProduction()` only; a mis-named prod-like environment skips them**
`WebApplicationBuilderExtensions.cs:33,72` and `ServiceCollectionExtensions.cs:27`. The FrontendUrl, connection-string, and (pre-existing) JWT-secret guards all fire only when `builder.Environment.IsProduction()` — i.e. only when `ASPNETCORE_ENVIRONMENT` is exactly `"Production"` (case-insensitive). The SMTP guard, by contrast, fires for **any** non-Development env (`ServiceCollectionExtensions.cs:63` `else` branch). So the guard set is internally inconsistent: an internet-facing deploy that sets `ASPNETCORE_ENVIRONMENT=Staging` (or a typo like `Prod`/`Prodction`) **boots past** the FrontendUrl/connection-string/JWT guards with blank values — broken magic links, a zero-origin CORS policy, 500s on first DB hit, and no JWT secret. BA-1/BA-2 extended the narrow `IsProduction()` pattern to two more guards rather than closing the gap. *Reachability caveat that keeps this out of High:* (a) `ASPNETCORE_ENVIRONMENT` **unset** defaults to `Production`, so a bare deploy is safe; (b) the truly dangerous dev shortcuts (auth bypass, `devLink`, email-file) are `IsDevelopment()`-gated and stay **off** in Staging; (c) a blank JWT secret configures no auth scheme, so authorized endpoints return 401 (fail-closed), not a bypass. Net effect is a half-broken boot on a mis-named env, not a data/auth breach. **Fix:** scope the FrontendUrl / connection-string / JWT guards to `!builder.Environment.IsDevelopment()` (matching the SMTP guard), or gate on an explicit allowlist of deployed env names, so every non-dev environment fails loud identically.

**S-M2 — Forwarded-header wildcard rejection is not airtight for IPv6 / split ranges**
`Program.cs:58-64`. The network guard rejects only the literal strings `"*"` and `"0.0.0.0/0"`. It does **not** reject the IPv6 all-networks CIDR `"::/0"`, nor equivalent full-coverage configurations expressed as two halves (`0.0.0.0/1` + `128.0.0.0/1`) or an over-broad prefix. `IPNetwork.Parse("::/0")` succeeds and is added to `KnownIPNetworks`, so on a dual-stack deploy an operator who trusts `::/0` re-opens exactly the `X-Forwarded-For` spoofing hole the guard exists to prevent — any IPv6 client could forge its client IP and defeat the magic-link limiter. The code and its comment advertise airtight wildcard rejection, so this is a real gap between claim and behaviour. Likelihood is low (an operator must actively type `::/0`), which keeps it Medium rather than High. **Fix:** reject any network whose prefix length is `0` (both `IPAddress.Parse(...)`-derived families) rather than string-matching two literals — e.g. after `IPNetwork.Parse`, throw if `PrefixLength == 0`; optionally reject unspecified/`::` and `0.0.0.0` proxy addresses too.

### Low / Hardening

**S-L1 — Malformed (non-wildcard) proxy/CIDR value throws a raw `FormatException` instead of a key-named error**
`Program.cs:55,66`. A typo'd value (`"10.0.0"`, `"not-an-ip"`, `"10.0.0.0/33"`) makes `IPAddress.Parse` / `IPNetwork.Parse` throw a bare `FormatException` at startup. This is still **fail-closed** (the app refuses to start — good), and the exception does not echo the value, but it doesn't name the offending config key the way the sibling guards do, so a deploy-time misconfig is harder to diagnose. **Fix:** wrap the two parse loops in a `try/catch (FormatException)` that rethrows `InvalidOperationException` naming `ForwardedHeaders:KnownProxies` / `KnownNetworks` (still without echoing the bad value).

**S-L2 — Recipient email logged at Information on every send (pre-existing PII, not introduced by this diff)**
`EmailService.cs:17` `logger.LogInformation("Sending email to {Email}", to)`. Every magic-link request writes the recipient address to logs at Information. This is PII over-logging and predates B-7; I flag it only because BA-4's log-hygiene pass touched adjacent logging config. It is not a config leak this sweep must fix — **defer to B-8** (or hash/redact the local-part). No token/link/body is logged, so this is low severity.
