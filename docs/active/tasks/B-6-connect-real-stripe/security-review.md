# Tidansu — Security Review
**Date:** 2026-07-12
**Scope:** B-6 (Connect real Stripe) — uncommitted working-tree diff. Billing seam:
`StripeBillingService`, `DisabledBillingService`, `ProcessedStripeEventStore`,
`ServiceCollectionExtensions` (DI selection), `StripeSettings`, `BillingController`,
`HandleStripeWebhookCommandHandler`, `ChangePlanCommandHandler`, `UserService`,
`TidansuDbContext`, `ErrorHandlingMiddleware`, `useSessionStore.ts`.
**Type:** Findings report only — no code changes made. `dotnet build` green (12 pre-existing NU1903 advisory warnings, unrelated to B-6).

**Overall:** The webhook-trust seam is well built and the four highest-risk controls the task set out to enforce are genuinely implemented: signature-verified-first, raw-body pass-through, server-set `ClientReferenceId` as the sole account authority, atomic insert-first idempotency, and a Production fallback (`DisabledBillingService`) that can never hand out free Pro. No forged-webhook → free-Pro path exists in a correctly-configured deployment. The one real exposure is a **configuration gap**: `WebhookSecret` — the bearer credential that is the *sole* gate on the anonymous webhook — is excluded from both `IsConfigured` and the fail-loud startup guard, so a deployment can boot Stripe-enabled with an empty signing secret, which is forgeable. Two correctness issues (premature Pro grant on unpaid sessions; non-atomic claim-vs-mutate) round out the list.

## What's already done right
- **Signature is verified first and nothing mutates on failure.** `HandleWebhookAsync` calls `EventUtility.ConstructEvent(payload, signature, WebhookSecret)` as its first statement (`StripeBillingService.cs:143`); any `StripeException` is turned into a `ValidationException` → 400 via `ErrorHandlingMiddleware.cs:15`, before any switch/lookup/mutation. Forged/missing signature changes no state.
- **Raw body preserved.** `BillingController.Webhook` reads `Request.Body` via `StreamReader` (`BillingController.cs:19-21`), forwards the `Stripe-Signature` header, and uses no model-bound DTO. `Program.cs` runs no body-buffering/parsing middleware ahead of the controller (`ForwardedHeaders` → `ErrorHandling` → static files → routing), so the exact bytes reach `ConstructEvent`.
- **No spoofable account mapping.** `checkout.session.completed` resolves the account solely from `session.ClientReferenceId` (server-set to `user.Id` at checkout, `StripeBillingService.cs:102,204`); it never trusts `CustomerEmail`. Later events resolve via stored `StripeSubscriptionId` (`:230,:255`). No IDOR / cross-account upgrade path.
- **Atomic idempotency, claimed before mutation.** `ProcessedStripeEventStore.TryMarkProcessedAsync` is insert-then-`SaveChanges`, catching `DbUpdateException` on the unique PK (`ProcessedStripeEventStore.cs:15-33`) — no read-then-write race. Every mutating event is gated on it *before* touching plan state (`StripeBillingService.cs:155,160,165`). Duplicate deliveries are 200 no-ops.
- **Production can never resolve `DirectBillingService`.** DI selection (`ServiceCollectionExtensions.cs:105-127`): `Enabled && IsConfigured` → Stripe; `Enabled && !IsConfigured` → **throws at startup** naming the missing `StripeSettings__*` key without echoing values (`:110-115`); `!Enabled` in Development → `DirectBillingService`; `!Enabled` in Production → `DisabledBillingService`. `DirectBillingService` (free flip) is unreachable outside Development.
- **`DisabledBillingService` never grants Pro** — upgrade throws `BillingUnavailableException` (→ 503 generic, `ErrorHandlingMiddleware.cs:141-160`); downgrade is a safe no-op keeping data (`DisabledBillingService.cs:17-35`).
- **Downgrade integrity.** Free is reached only via the verified `customer.subscription.deleted` event (`:248-268`); it flips `Plan` and clears cancel flags but deletes no data. `invoice.payment_failed` deliberately does not downgrade (dunning grace, `:169-173`). Read-only-over-cap enforcement (`PlanPolicy`) is untouched.
- **No secrets in source.** `appsettings.Development.json` carries blank `SecretKey`/`WebhookSecret`/`ProPriceId` placeholders; no `sk_`/`whsec_` literal anywhere in tracked source. Logs record event type + user id + subscription/session id only — never raw payload, signature, or PII; the signature-failure log emits `ex.Message` only.
- **Schema hygiene.** Indexes on `User.StripeSubscriptionId`/`StripeCustomerId` back the webhook lookups (`TidansuDbContext.cs:46-47`); the mutate path uses the tracked `UserManager` entity (not `AsNoTracking`), so updates persist.

## Security findings

### High
**S-H1 — `WebhookSecret` omitted from `IsConfigured` and the fail-loud guard → forgeable webhook on a plausible misconfig**
`StripeSettings.cs:23-26` (`IsConfigured = Enabled && SecretKey && ProPriceId`) and `ServiceCollectionExtensions.cs:110-115` (guard checks only `SecretKey` and `ProPriceId`). The webhook signing secret — the *only* gate on the `[AllowAnonymous]` webhook — is never required. A deployment set with `Enabled=true`, `SecretKey`, `ProPriceId`, but an **empty** `StripeSettings__WebhookSecret` (blank env var, copy-paste slip) boots cleanly and registers `StripeBillingService`. `EventUtility.ConstructEvent(payload, signature, "")` (`StripeBillingService.cs:143`) then verifies against an HMAC-SHA256 keyed with the empty string — a key the attacker also knows. **Exploit:** attacker computes a valid `Stripe-Signature` over a forged `checkout.session.completed` body with `client_reference_id` = any/attacker user id and POSTs it to `/api/billing/webhook` → `OnCheckoutCompletedAsync` grants that account Pro for free (`:196-219`). This is exactly the "forged webhook = free Pro" Critical the task calls out; it is gated only by an operator leaving one key blank, which the fail-loud pattern was meant to prevent. (If `WebhookSecret` is `null` rather than `""`, `ConstructEvent` instead throws → every webhook 500s and paying users are never upgraded — a silent revenue/delivery failure. Either way the key must be guarded.) Rated High because it requires a specific misconfiguration; treat as **Critical-if-empty**. **Fix:** add `!string.IsNullOrWhiteSpace(WebhookSecret)` to `IsConfigured`, and add `WebhookSecret` to the missing-keys list in the startup guard so `Enabled && !IsConfigured` fails loud naming `StripeSettings__WebhookSecret`.

### Medium
**S-M1 — Pro granted on `checkout.session.completed` without checking `session.PaymentStatus`**
`StripeBillingService.cs:196-219`. `OnCheckoutCompletedAsync` sets `Plan = Pro` for any verified `checkout.session.completed`, regardless of `session.PaymentStatus`. For card payments this is `"paid"`, but the B-5/Poland launch is likely to enable delayed/async methods; for those Stripe fires `checkout.session.completed` with `payment_status = "unpaid"` and the subscription `incomplete`. The account is promoted to Pro before money settles, and if the initial payment ultimately fails the only correction is the later `customer.subscription.deleted` after dunning — a free Pro window in between. **Fix:** in `OnCheckoutCompletedAsync`, gate the grant on `session.PaymentStatus == "paid"` (or `"no_payment_required"`), or drive the grant from `invoice.paid` / `customer.subscription.created` (active) for delayed methods. Keep the idempotency claim, but only mutate when payment is confirmed.

**S-M2 — Non-atomic idempotency claim vs. plan mutation → paying user can be left un-upgraded**
`StripeBillingService.cs:155` + `196-216`. `AlreadyProcessedAsync` commits the ledger row (its own `SaveChangesAsync`, `ProcessedStripeEventStore.cs:24`) *before* `OnCheckoutCompletedAsync` mutates the user in a separate `UserManager.UpdateAsync` commit. If the mutation fails (transient DB error, concurrency, `UpdateAsync` returning failure → `ValidationException`), the event is already recorded as processed; the handler throws → 500 → Stripe retries → `AlreadyProcessedAsync` now returns `true` → the retry is skipped and acked 200. Net: Stripe's own retry safety net is defeated for the one delivery that mattered, and a paying customer is permanently left on Free until manual intervention. **Fix:** make claim + mutation one unit of work — either wrap both in a single transaction/`SaveChanges`, or record the event as processed only *after* the user mutation succeeds (still atomic via the unique PK), or on mutation failure remove the just-inserted ledger row before rethrowing so the retry re-processes.

### Low / Hardening
**S-L1 — Anonymous webhook has no rate limit or explicit body-size cap**
`BillingController.cs:14-25` is `[AllowAnonymous]` and outside any rate-limiter policy; each request reads the full body into a string (`ReadToEndAsync`) and computes an HMAC before rejecting. An attacker can flood it with large-bodied, bad-signature requests for cheap CPU/allocation pressure. Low impact (signature check is inexpensive, no DB touched pre-verification). **Fix (defense-in-depth):** apply a dedicated lenient rate-limit policy tuned for Stripe's burst behaviour and/or a small `[RequestSizeLimit]` on the action so oversized bodies are rejected before buffering. Do not add per-IP limits tight enough to drop legitimate Stripe bursts.

**S-L2 — `ConstructEvent` API-version-mismatch behaviour is implicit**
`StripeBillingService.cs:143` uses the 3-arg `ConstructEvent`. If a future maintainer switches to an overload that throws on API-version mismatch, a Stripe account/library version drift would 400 every webhook (silent upgrade outage). Not exploitable; note only. **Fix:** pin the Stripe API version explicitly and add a comment that signature verification must not become version-fragile.

**S-L3 — Client trusts persisted `plan` in localStorage (pre-existing, not B-6)**
`useSessionStore.ts:75-78,124-135` derive `isPro`/`caps` from a localStorage-persisted plan the user can edit; the optimistic flip in `setPlan` is reverted on redirect but the store still trusts client state for UI gating. Server-side `PlanPolicy` remains the real authority for mutations, so this is UI-only and out of B-6's scope — noted so it is not mistaken for a new regression. No server trust of client plan was introduced by B-6.

## Verification checklist
- [ ] **S-H1:** In a Production-like env set `StripeSettings__Enabled=true`, `__SecretKey`, `__ProPriceId`, and `__WebhookSecret=""` (empty). Confirm the app currently **boots** (the bug) and that after the fix it **fails loud** naming `StripeSettings__WebhookSecret`. Then, against the unpatched build, sign a forged `checkout.session.completed` with an empty-string HMAC secret and `client_reference_id` of a test user, POST it to `/api/billing/webhook`, and confirm it grants Pro — then confirm the patched build rejects it (400).
- [ ] **S-M1:** With a delayed/async test payment method (e.g. a redirect method that leaves `payment_status=unpaid`), complete Checkout and confirm the account is **not** promoted to Pro until payment is confirmed.
- [ ] **S-M2:** Force `UserManager.UpdateAsync` to fail once during `checkout.session.completed` (e.g. inject a transient error), then have Stripe resend the event (`stripe events resend <id>`). Confirm the retry actually upgrades the user (i.e. the first failure did not permanently consume the idempotency claim).
- [ ] **Regression (must stay green):** bad/missing `Stripe-Signature` → 400, no plan change; duplicate valid `checkout.session.completed` → second is a 200 no-op; `!Enabled` in Production resolves `DisabledBillingService` and an upgrade returns 503 with the account staying Free; downgrade → `cancel_at_period_end=true`, Pro retained until `ProAccessUntil`, Free only on `customer.subscription.deleted`.
